# Vibecoding 方法论与标准工作流手册

更新时间：2026-03-27

这份手册不是在教你“怎么让 AI 多写一点代码”，而是在教你如何把 AI 纳入一个可控、可回滚、可扩展的个人开发流程。

它的核心立场只有一句话：

> AI 不是主程，AI 是施工队。  
> 人必须担任架构师、产品经理、验收者和风险控制者。

---

## 1. 先讲结论

### 1.1 Vibecoding 的本质

Vibecoding 真正的问题从来不是“AI 会不会写”，而是下面这 5 件事有没有被管住：

1. 目标是否清楚，不清楚就会越写越偏。
2. 边界是否清楚，不清楚就会误改、漏改、乱改。
3. 上下文是否受控，不受控就会 token 爆炸、记忆错乱、重复劳动。
4. 代码是否可回滚，不可回滚就不敢快，敢快也会翻车。
5. 排错是否有路径，没有路径就会沦为“不断试一试”。

### 1.2 原始材料可提炼成的统一原则

1. 先目标后编码。
2. 先边界后实现。
3. 先模块后文件。
4. 先分层后改动。
5. 先蓝图后施工。
6. 先规格后联调。
7. 单任务单上下文。
8. 小步提交，可随时回滚。
9. 截图、目录树、流程图优先于模糊描述。
10. 最小闭环优先于大而全。
11. 定位问题先分层，再追数据流。
12. 只修局部，不借机全局重构。
13. AI 完工后先自检，再由人复核。
14. 涉及密钥、权限、支付、生产数据时必须人工兜底。

### 1.3 个人开发者最容易翻车的 7 个点

1. 一上来让 AI “做一个完整产品”。
2. 功能、界面、数据库、接口全塞在一次会话里。
3. 没有 Git 小步提交，出了问题只能整体回退。
4. 没有规则文件，每轮都重新解释项目规范。
5. 需求变了但规格没变，导致新旧约束混杂。
6. 报错后不分层，直接让 AI “修一下”。
7. 把工具原生能力当不存在，提前做了很多不必要准备。

## 2. Vibecoding 方法论总纲

### 2.1 四层角色分工

人在 vibecoding 里至少要同时扮演 4 个角色：

| 角色 | 你负责什么 | 不能外包给 AI 的部分 |
| --- | --- | --- |
| 产品经理 | 目标用户、价值、MVP 边界 | 做不做、先做什么、不做什么 |
| 架构师 | 模块划分、分层、接口、依赖关系 | 边界定义、复杂度控制 |
| 施工指挥 | 本轮改什么、不改什么、怎么验证 | 执行顺序、风险控制 |
| 验收者 | 测试、回归、代码审查、上线决策 | 最终是否通过 |

AI 负责的是：

1. 按你的边界快速施工。
2. 在受控上下文里产出代码、文档、测试和蓝图。
3. 帮你做自检、回归、重构建议和方案比较。

### 2.2 从“脚本思维”升级到“产品思维”

小脚本时代，AI 可以直接写。

产品时代，必须先回答 4 个问题：

1. 这个产品的最小闭环是什么。
2. 这个闭环由哪些模块组成。
3. 这些模块如何分层，如何通信。
4. 每一轮开发只允许动哪一块。

如果这 4 个问题没有先回答，vibecoding 就会退化成：

- 大提示词碰运气
- 改一处坏三处
- 上下文越来越脏
- 功能越做越难加

### 2.3 你真正需要的不是“大 prompt”，而是“小而硬的约束”

高质量 vibecoding 依赖的不是一次性长提示词，而是 4 种长期约束：

1. `项目目标约束`
2. `架构边界约束`
3. `编码与命名约束`
4. `测试与提交流程约束`

这些约束应该沉淀到规则文件，而不是每轮现说。

## 3. 标准 Vibecoding 9 步工作流

下面这 9 步就是标准 SOP。  
新项目和老项目都能用，差别只在于产物做轻还是做重。

### 第 1 步：定北极星

**目标**

把“想做个什么”压缩成一个可执行目标。

**动作**

1. 写一句话目标。
2. 写清楚目标用户是谁。
3. 写清楚成功标准。
4. 写清楚第一版不做什么。

**产物**

- 一句话目标卡
- 成功标准
- 非目标清单

**给 AI 的 Prompt**

```md
你现在扮演产品经理。先不要写代码。
请根据以下想法，帮我压缩成一个可执行的一句话目标，并补全：
1. 目标用户
2. 核心价值
3. 成功标准
4. 第一版明确不做什么

要求：
- 只保留最小闭环
- 不要自动扩展成平台级产品
- 输出结果用 Markdown 表格
```

**检查点**

1. 目标是否能在一句话里说清楚。
2. 成功标准是否可验证。
3. 是否明确写了“不做什么”。

**常见翻车点**

1. 把“长期愿景”当成“第一版目标”。
2. 没写非目标，导致 AI 自动加戏。

---

### 第 2 步：立边界

**目标**

明确哪些地方可以让 AI 自主做，哪些地方必须人工审核。

**动作**

1. 判断是否涉及用户数据、支付、权限、生产环境。
2. 决定是否允许自动执行命令、自动安装依赖、自动改配置。
3. 明确哪些目录、文件、环境变量属于敏感区。

**产物**

- 风险边界卡
- 敏感目录清单
- 自动化权限清单

**给 AI 的 Prompt**

```md
请作为技术负责人，帮我为这个项目建立风险边界。
先不要写代码，只输出：
1. 敏感数据与敏感操作清单
2. 哪些命令允许 AI 自动执行
3. 哪些文件或目录禁止 AI 自动修改
4. 哪些改动必须人工复核

请按“低风险 / 中风险 / 高风险”分级。
```

**检查点**

1. 是否明确列出 `.env`、支付、鉴权、生产配置等高风险区域。
2. 是否有“必须人工复核”的清单。

**常见翻车点**

1. 把 API Key 写进 prompt 或前端代码。
2. 没有权限边界，导致 AI 顺手改了锁文件、部署脚本或生产配置。

---

### 第 3 步：拆 MVP

**目标**

把产品想法缩成一个最小可验证闭环。

**动作**

1. 只保留“用户第一次得到价值”的必要链路。
2. 删除第二阶段、第三阶段功能。
3. 为第一版定义一个 Demo 级完成标准。

**产物**

- MVP 清单
- 推迟功能清单

**给 AI 的 Prompt**

```md
请把下面这个需求缩减成一个 MVP。
要求：
- 只保留最小可验证闭环
- 删除会显著增加复杂度但不影响第一次验证价值的功能
- 输出“本版必须有 / 本版不做 / 下一版再做”
- 说明每个删除项为什么应当后置
```

**检查点**

1. 有没有删掉“多角色、多权限、多端同步、复杂搜索、报表、后台系统”这类会炸复杂度的部分。
2. MVP 是否能在 1 到 3 个核心模块内闭环。

**常见翻车点**

1. “顺手加一下”导致架构从单机变多租户。
2. 为未来扩展提前实现过多抽象。

---

### 第 4 步：拆模块

**目标**

让 AI 每次只在一个清晰边界内施工。

**动作**

1. 先画目录树，不先写文件。
2. 把大功能拆成单一职责模块。
3. 明确每个模块的输入、输出和依赖。

**产物**

- 模块清单
- 目录树
- 模块职责表

**给 AI 的 Prompt**

```md
你现在扮演资深架构师。
请基于这个 MVP 输出一个标准目录树，并解释每个模块的职责。

要求：
- 单一职责
- 命名统一
- 可独立测试
- 尽量减少跨模块耦合
- 先不要生成实现代码
```

**检查点**

1. 每个模块是否能用一句话说明职责。
2. 是否存在“万能 utils”或“超级 service”。
3. 是否能把某个模块单独拿出来测试。

**常见翻车点**

1. 按页面拆了界面，却没拆业务模块。
2. 把所有东西都丢进 `lib`、`utils`、`services`。

---

### 第 5 步：做三层分层

**目标**

把改动范围精确限制在界面层、逻辑层、数据层中的某一层或几层。

**动作**

1. 标注哪些代码属于界面层。
2. 标注哪些代码属于逻辑层。
3. 标注哪些代码属于数据层。
4. 明确本轮只动哪一层。

**产物**

- 三层映射表
- 本轮改动边界说明

**给 AI 的 Prompt**

```md
请把这个模块拆成三层：
1. 界面层
2. 逻辑层
3. 数据层

要求：
- 用三列表格输出
- 标注每层职责
- 标注本轮如果只改 UI，哪些文件不该动
- 标注跨层依赖
```

**检查点**

1. UI 改动能否做到不碰数据层。
2. 业务规则是否都留在逻辑层，而不是写进组件或 SQL。

**常见翻车点**

1. 在组件里直接写复杂规则。
2. 在数据层偷偷处理业务判断。

---

### 第 6 步：先出蓝图

**目标**

在动手之前，先确认数据流和模块依赖是闭环的。

**动作**

1. 先让 AI 输出 Mermaid 蓝图。
2. 检查用户路径、数据路径、错误路径。
3. 让 AI 再给出执行计划。

**产物**

- Mermaid 数据流图
- 执行计划
- 影响范围说明

**给 AI 的 Prompt**

```md
先不要写代码。
请用 Mermaid 画出以下功能的：
1. 用户操作流
2. 模块调用链
3. 数据输入输出路径
4. 失败或异常路径

然后补充一个执行计划：
- 分几步做
- 每步修改哪些文件
- 每步影响哪些模块
- 每步完成后怎么验证
```

**检查点**

1. 数据流是否闭环。
2. 错误路径有没有被画出来。
3. 执行计划是否能拆成小步。

**常见翻车点**

1. 没蓝图直接开工。
2. 蓝图只画成功路径，不画失败路径。

---

### 第 7 步：先写规格

**目标**

在联调前把接口契约写死，减少模块错位。

**动作**

1. 定义输入输出字段。
2. 定义字段类型、校验规则、错误返回。
3. 定义命名规范、测试标准、边界场景。

**产物**

- 接口契约
- 字段定义
- 错误码或错误返回规范

**给 AI 的 Prompt**

```md
先不要写实现。
请为下面这个模块输出规格书，包含：
1. 输入参数
2. 返回结构
3. 字段类型与约束
4. 校验规则
5. 错误返回
6. 边界情况
7. 对应测试点

如果字段未定义，请先提出候选方案，不要自行假定。
```

**检查点**

1. 有没有用明确字段名、类型和约束。
2. 错误返回是否可预期。
3. 测试点是否从规格反推出来。

**常见翻车点**

1. 口头约定接口，没有落文档。
2. 规格变了但代码和测试没同步。

---

### 第 8 步：小步施工

**目标**

把 AI 的执行粒度压缩到“单模块、单目标、可验证”。

**动作**

1. 一次只做一个模块或一个子任务。
2. 一个会话尽量只处理一种目标。
3. 改完立刻运行最小验证。
4. 通过后立即 Git 提交。

**产物**

- 小步提交记录
- 变更说明
- 最小验证结果

**给 AI 的 Prompt**

```md
只实现这个子任务，不做额外扩展：
[粘贴子任务描述]

约束：
- 只修改这几个文件：[文件列表]
- 不允许修改这些层或模块：[禁止改动列表]
- 如果发现更大范围重构机会，只记录建议，不要顺手改
- 完成后请输出：
  1. 改了什么
  2. 没改什么
  3. 需要我运行哪些验证
```

**检查点**

1. 改动是否局部。
2. 是否存在“顺手修一下”的范围蔓延。
3. 每次提交是否都能独立回滚。

**常见翻车点**

1. 一次让 AI 做太多。
2. 没验证就进入下一步。
3. 多个未提交改动混在一起。

---

### 第 9 步：排错与复盘

**目标**

把问题修在最小范围内，并把经验沉淀成长期规则。

**动作**

1. 先判断问题在界面层、逻辑层还是数据层。
2. 沿数据流逐环节追踪输入输出。
3. 只修问题文件和必要依赖。
4. 做回归验证。
5. 把这次经验沉淀到规则文件。

**产物**

- 调试记录
- 修复说明
- 回归结果
- 规则更新

**给 AI 的 Prompt**

```md
不要直接改代码，先排错。

请按以下顺序输出：
1. 先判断问题属于界面层、逻辑层还是数据层
2. 给出最可能的 3 个原因
3. 设计最短排查路径
4. 标出需要看的文件与日志
5. 如果确认根因，再给出最小修复方案

要求：
- 不要一上来全局搜索替换
- 优先单点修复
- 最后补充回归测试点
```

**检查点**

1. 是否先定位再修改。
2. 修复是否局限在问题域。
3. 是否更新了规则或防错清单。

**常见翻车点**

1. 一看到报错就全局乱改。
2. 修好了症状，没修根因。
3. 没回归测试，旧问题又回来。

## 4. 高频排错与防翻车清单

### 4.1 调试三步法

1. `定位层`
   先判断问题属于界面层、逻辑层还是数据层。
2. `追踪流`
   从用户操作一路追输入、状态、请求、返回、渲染。
3. `精准修`
   只修改出问题文件和必要依赖，不趁机重构全仓。

### 4.2 AI 写完后必须追加的 5 个问题

1. 这次改动最可能的边界情况是什么。
2. 有没有引入空值、未授权、重复提交、并发、类型不一致问题。
3. 哪些已有功能可能被影响。
4. 这次最小验证集是什么。
5. 如果要回滚，最小回滚单位是什么。

### 4.3 五条硬性纪律

1. 没有提交点，不进入下一轮。
2. 没有规则文件，不长期协作。
3. 没有规格，不跨模块联调。
4. 没有验证，不宣布完成。
5. 涉及真实数据和高权限，不开自动放行。

## 5. 提效 Tools / Skills 合集

这一部分的目标不是“列最多工具”，而是给个人开发者一套够用、稳定、低心智负担的组合。

### 5.1 我推荐的默认组合

#### 组合 A：稳定效率优先

- 主力代理：`Claude Code` 或 `Cursor`
- 补充执行：`GitHub Copilot coding agent`
- Git 护栏：`Git + 小步提交`
- 长期规则：`CLAUDE.md` 或 `.cursor/rules`
- 工具扩展：`MCP`
- 测试验证：项目自带测试框架 + lint/build

适合：

- 你希望在一个主工作台完成大部分工作。
- 你重视上下文治理和工程纪律。

#### 组合 B：终端流与回滚优先

- 主力代理：`Aider`
- 规划/文档：`Claude Code` 或 `Cursor`
- 规则文件：`AGENTS.md` 或项目文档
- 自动检查：`Aider auto lint/test`

适合：

- 你偏命令行工作流。
- 你特别重视 Git 回滚和小步迭代。

#### 组合 C：开源可控优先

- 主力代理：`Continue` 或 `Roo Code / Cline`
- 长期规则：`.continue/rules` 或 `AGENTS.md`
- 工具扩展：`MCP`
- 人工护栏：更严格的审批和目录限制

适合：

- 你希望更强可配置性。
- 你愿意自己承担更多工具治理成本。

### 5.2 核心 skills 合集

如果你使用支持 skills 的 agent，我建议第一批只装这几类。

#### 核心必备

| Skill | 作用 | 该在什么时候触发 |
| --- | --- | --- |
| `brainstorming` | 先把需求聊清楚，不直接施工 | 每次新需求开始时 |
| `writing-plans` | 把设计变成执行计划 | 设计确定后 |
| `systematic-debugging` | 强制走定位-验证流程 | 一出现 bug 或测试失败 |
| `test-driven-development` | 关键逻辑先锁测试 | 业务规则、算法、接口 |
| `verification-before-completion` | 防止“没验证就说完成” | 收尾前 |
| `requesting-code-review` | 主动拉审查 | 大任务完成后 |
| `receiving-code-review` | 规范处理 review 意见 | 收到评审反馈时 |

#### 进阶提效

| Skill | 作用 | 适合场景 |
| --- | --- | --- |
| `subagent-driven-development` | 把独立任务分派给子代理 | 中大型任务 |
| `dispatching-parallel-agents` | 并行多个独立侧任务 | 搜索、验证、文档、实现可分开时 |
| `using-git-worktrees` | 隔离高风险改动 | 重要分支或多方案试验 |
| `finishing-a-development-branch` | 收尾、合并、清理流程 | 功能完成后 |
| `find-skills` | 搜合适技能 | 发现能力缺口时 |

#### 领域增强

| Skill / 资源 | 作用 | 适合场景 |
| --- | --- | --- |
| `frontend-design` | 提升 UI 输出质量 | 前端页面和组件 |
| `figma-implement-design` | 从 Figma 到代码 | 设计还原 |
| `web-design-guidelines` | 用规则审查 Web 体验 | UI 审计 |

### 5.4 AI coding 工具推荐矩阵

| 工具 | 原生强项 | 对 vibecoding 的价值 | 适合谁 |
| --- | --- | --- | --- |
| `Cursor` | Rules、Codebase Indexing、PR Search、MCP、Background Agents | 持久规则、代码库理解、异步后台施工 | 想在编辑器里一站式工作的人 |
| `Claude Code` | `CLAUDE.md`、`/clear`、`/compact`、`/agents`、`/memory`、`/mcp`、hooks、subagents | 规划、上下文治理、任务拆分、受控执行 | 重视工程方法和代理治理的人 |
| `Aider` | repo map、ask/code/architect、auto lint/test、Git 自动提交、`/undo` | 回滚安全、小步施工、终端流高效率 | 偏 CLI 的个人开发者 |
| `GitHub Copilot` | coding agent、code review、custom instructions、`AGENTS.md` | 后台做 PR、做代码审查、沉淀仓库指令 | 已在 GitHub 流程里开发的人 |
| `Continue` | rules blocks、tool policies、MCP、多模型自定义 | 高可配置、开源、模型与工具可替换 | 想高度自定义的人 |
| `Roo Code / Cline` | 多模式代理、终端/编辑器联动、MCP | 开放、代理感强、可扩展 | 愿意自行管理更强自动化的人 |

### 5.5 各工具“本来就能做到什么”

这一节非常关键，因为它决定了你在项目开始前要不要做那么多准备。

#### Cursor

原生就能帮你做的事：

1. `Codebase Indexing`
   自动索引代码库并逐步增量更新。
2. `PR Search`
   通过历史 PR 帮你理解代码演进。
3. `Project Rules`
   把项目规范长期挂在上下文里。
4. `MCP`
   直接连接外部工具和数据源。
5. `Background Agents`
   在隔离的远程环境异步跑任务。

所以你通常可以少做的准备：

1. 不用手写一份冗长代码库总览。
2. 不用每轮都重贴同样的规范。
3. 不用所有任务都在一个实时会话里做。

#### Claude Code

原生就能帮你做的事：

1. `CLAUDE.md memory`
   可以把项目规则和常用命令长期记住。
2. `/clear`、`/compact`
   可以主动清理和压缩上下文。
3. `/agents` 与 `subagents`
   可以把探索、计划、执行分开。
4. `hooks`
   可以把格式化、日志、敏感文件拦截做成确定性规则。
5. `MCP`
   可以直接把外部工具接进来。

所以你通常可以少做的准备：

1. 不用为了“怕记不住”而维护很厚的手工说明文档。
2. 不用所有规则都写进 prompt。
3. 不用所有任务都在主对话上下文里做。

#### Aider

原生就能帮你做的事：

1. `repo map`
   自动给模型提供仓库结构和关键符号。
2. `ask / code / architect`
   自带讨论、编辑和架构模式。
3. `auto lint/test`
   改完自动跑检查。
4. `Git integration`
   自动提交、脏文件隔离、`/undo` 回滚。

所以你通常可以少做的准备：

1. 不用手写太多“仓库结构说明”。
2. 不用手工搭一套最基础的回滚机制。
3. 不用每次改完再人工记得去 lint/test。

#### GitHub Copilot

原生就能帮你做的事：

1. `coding agent`
   可以后台在 GitHub 上异步做任务并开 PR。
2. `code review`
   可以自动参与审查。
3. `repository custom instructions`
   支持 `.github/copilot-instructions.md`。
4. `path-specific instructions`
   支持 `.github/instructions/*.instructions.md`。
5. `AGENTS.md`
   最近目录的 `AGENTS.md` 会参与约束。

所以你通常可以少做的准备：

1. 不用每次口头解释仓库规范。
2. 不用全靠手工把 AI 产物搬成 PR。
3. 不用把每个任务都拉回本地同步处理。

#### Continue

原生就能帮你做的事：

1. `Rules`
2. `Agent Mode`
3. `Tool Policies`
4. `MCP`
5. 多模型和系统消息定制

所以你通常可以少做的准备：

1. 不用把规则散落在对话里。
2. 不用为每种任务切换不同工具链。

#### Roo Code / Cline

原生就能帮你做的事：

1. 多模式代理工作流
2. 终端、浏览器、编辑器协同
3. MCP 扩展

所以你通常可以少做的准备：

1. 不用为了复杂任务单独再拼太多小工具。

但必须额外注意：

1. 自动化越强，权限治理越重要。
2. 不能因为它“会做很多”，就放弃风险边界。

## 6. 最小准备清单与“可省略准备”边界

### 6.1 必须由人先定义的内容

这部分不能偷懒，也不能完全交给 AI。

| 必须由人先定义 | 为什么不能省 |
| --- | --- |
| 产品目标与目标用户 | 没有北极星，AI 会自动扩展需求 |
| 第一版范围与“不做什么” | 没有非目标，复杂度会膨胀 |
| 数据、权限、安全红线 | 工具可以执行，但不能替你承担风险 |
| 核心模块边界 | 不定义边界，AI 会跨模块乱改 |
| 完成定义与验收标准 | 不定义“何时算完成”，就会无限修改 |
| 关键接口契约与高风险字段约束 | 模块协作最容易在这里错位 |

### 6.2 很多时候可以不准备得那么重

如果你是单人开发者，很多项目不必一开始就写完整 PRD、完整架构文档、完整测试矩阵。

你至少准备下面这 6 项就够启动：

1. 一句话目标
2. 目标用户
3. 第一版只做什么
4. 第一版明确不做什么
5. 哪些区域属于高风险禁区
6. 本轮只动哪些模块

### 6.3 轻量启动版模板

```md
# 项目启动卡

一句话目标：

目标用户：

第一版只做：

第一版不做：

高风险禁区：

本轮改动范围：

完成定义：
```

### 6.4 什么时候必须从“轻量准备”升级到“重准备”

出现以下任一情况，就要从轻量卡片升级到更正式的 PRD、蓝图和规格：

1. 有登录、权限、支付、计费。
2. 涉及真实用户数据。
3. 需要多人协作。
4. 需要跨多个核心模块联动。
5. 有上线、审计或合规要求。
6. 未来 1 到 2 个迭代内一定会继续扩展。

## 7. Prompt 模板库

下面这套模板不是让你一股脑全用，而是按阶段抽取。

### 7.1 目标定义模板

```md
你现在扮演产品经理，不要写代码。
请把下面这个想法整理成一个可执行目标。

输出结构：
1. 一句话目标
2. 目标用户
3. 核心价值
4. 成功标准
5. 第一版不做什么

要求：
- 只保留最小闭环
- 不要自动扩展需求
```

### 7.2 模块拆分模板

```md
你现在扮演资深架构师。
基于这个 MVP，输出标准目录树和模块职责表。

要求：
- 单一职责
- 低耦合
- 可独立验证
- 命名统一
- 先不要写实现代码
```

### 7.3 三层分层模板

```md
请把这个功能拆成：
1. 界面层
2. 逻辑层
3. 数据层

请输出：
- 每层职责
- 每层涉及文件
- 本轮允许改动的层
- 本轮禁止改动的层
```

### 7.4 Mermaid 蓝图模板

```md
先不要写代码。
请用 Mermaid 输出：
1. 用户操作流程
2. 模块调用关系
3. 数据输入输出路径
4. 失败路径

然后给出一个执行计划：
- 分几步做
- 每步改哪些文件
- 每步的验证方式
```

### 7.5 规格书模板

```md
先不要实现。
请为这个模块生成规格书，包含：
1. 输入参数
2. 返回结构
3. 字段类型
4. 校验规则
5. 错误返回
6. 边界情况
7. 对应测试点
```

### 7.6 单模块开发模板

```md
只做这个子任务，不要做额外扩展：
[任务描述]

约束：
- 只修改这些文件：[文件列表]
- 不允许修改这些层或模块：[禁止列表]
- 如果发现额外优化机会，只列建议，不要顺手改

完成后请输出：
1. 改了什么
2. 没改什么
3. 需要运行哪些验证
4. 还剩什么风险
```

### 7.7 排错模板

```md
不要直接改代码，先排错。

请按顺序输出：
1. 先判断问题所在层级
2. 给出最可能的根因列表
3. 给出最短排查路径
4. 标出需要查看的文件、日志或请求链
5. 给出最小修复方案
6. 给出回归测试点
```

### 7.8 自检模板

```md
请对你刚才的改动做一次严格自检。

输出：
1. 这次改动覆盖了什么需求
2. 没覆盖什么
3. 最可能遗漏的边界情况
4. 可能影响的旧功能
5. 需要运行的最小验证集
6. 如果回滚，最小回滚单位是什么
```

### 7.9 规则文件生成模板

```md
请根据这个项目的技术栈和开发方式，帮我生成一份适合 AI coding 工具长期加载的项目规则。

必须包含：
1. 项目目标与范围
2. 目录职责
3. 命名规范
4. 分层原则
5. 不允许触碰的区域
6. 测试与提交流程
7. 常用命令
8. 安全约束

输出为 Markdown，适合放入项目规则文件。
```

## 8. 附录：推荐规则文件结构

下面不是强制标准，而是一套很稳的起步方式。

### 8.1 `CLAUDE.md`

适合放：

1. 项目目标
2. 目录职责
3. 常用命令
4. 测试要求
5. 禁改区域
6. 安全约束

示例：

```md
# Project Context

## Goal
- 这是一个面向个人开发者的轻量项目，优先快速验证而非平台化扩展。

## Architecture Rules
- 界面层不得直接访问数据库。
- 业务规则必须收敛在逻辑层。
- 数据层只做存取与持久化，不做页面判断。

## Safety Rules
- 不要修改 `.env`、锁文件、部署配置，除非我明确要求。
- 涉及认证、支付、生产数据的改动必须先给方案，再执行。

## Validation
- 每次修改后至少说明需要运行哪些测试。
```

### 8.2 `.cursor/rules/project.mdc`

适合放：

1. 目录边界
2. 命名风格
3. 文件组织约束
4. 测试与提交流程

### 8.3 `AGENTS.md`

适合放：

1. 仓库导航建议
2. 构建、测试、lint 命令
3. PR 和 commit 要求
4. 模块边界和禁区

### 8.4 `.github/copilot-instructions.md`

适合放：

1. 仓库级编码偏好
2. 测试流程
3. 不允许的改动方式
4. 审查前必须完成的检查

## 9. 最后给个人开发者的执行建议

如果你现在就要开始 vibecoding，不要先做一大堆文档。  
先做下面这件事：

1. 写一张启动卡。
2. 让 AI 帮你拆 MVP。
3. 让 AI 画模块树和三层表。
4. 为第一轮任务写出“只改哪里、不改哪里”。
5. 开始小步施工。
6. 每完成一步就验证并提交。

你真正要追求的不是“AI 一次把全项目做完”，而是：

> 让 AI 在你定义好的边界里，持续稳定地产生正确的小结果。

这才是个人开发者最实用、最稳、最能长期提效的 vibecoding 工作流。
