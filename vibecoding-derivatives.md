## 版本一：适合发文章 / 公众号的表达版

# Vibecoding 不是让 AI 替你编程，而是让你学会带着 AI 做产品

很多人理解的 vibecoding，还是一句话需求丢给 AI，然后等它吐出一堆代码。

这当然能做出一点东西，但也很容易进入另一个状态：功能越写越乱、上下文越聊越脏、改一个地方坏三个地方，最后你明明用了最强的 AI，却依然在低水平反复返工。

所以，真正决定 vibecoding 水平的，从来不是你会不会写 prompt，而是你有没有一套系统性的工作方法。

我更认同这样一句话：

> AI 不是主程，AI 是施工队。  
> 人必须担任架构师、产品经理、验收者和风险控制者。

当你把这个角色关系摆正，很多问题都会瞬间清楚。

## 一、为什么很多人 vibecoding 会翻车

大多数翻车都不是因为模型不够强，而是因为流程失控。

最常见的几个问题是：

1. 一上来就让 AI “做一个完整产品”。
2. 没有先拆模块，结果所有功能都缠在一起。
3. 没有分层，UI、业务逻辑、数据库到处互相渗透。
4. 没有蓝图，先写再说，最后只能靠返工补理解。
5. 没有规格，模块之间靠猜，联调时开始错位。
6. 没有 Git 小步提交，改坏了也不敢回退。
7. 一报错就让 AI “修一下”，但自己并不知道问题到底在哪一层。

这些问题加在一起，会让 vibecoding 退化成一种表面高效、实际低质的开发方式。

所以，真正的分水岭不在于“你有没有用 AI”，而在于“你有没有用系统思维去驾驭 AI”。

## 二、vibecoding 的核心，不是写代码，而是控边界

如果只写一个静态页面，AI 的确可以直接上手。

但只要你做的是一个真实产品，你就必须先回答下面几个问题：

1. 这个产品第一版到底要解决什么问题？
2. 最小可验证闭环是什么？
3. 这个闭环由哪些模块组成？
4. 每个模块属于界面层、逻辑层还是数据层？
5. 本轮开发允许动哪些区域，不允许动哪些区域？

本质上，vibecoding 不是“把开发交给 AI”，而是“把 AI 放进一个你定义好的边界里，让它稳定地产出正确的小结果”。

## 三、四个最关键的架构思维

### 1. 模块拆解

先拆模块，再让 AI 分块施工。

不要把“用户登录、订单处理、消息通知、页面渲染、数据库读写”混成一个任务。  
正确做法是：你先定义边界，AI 再进入单模块施工。

一句话概括就是：

> 你当总设计师，AI 当施工队。

### 2. 三层分层

任何稍微像样一点的产品，都应该先分成三层：

- `界面层`：页面、按钮、动画、交互
- `逻辑层`：规则、计算、状态、流程判断
- `数据层`：数据库、接口、持久化、外部依赖

三层分清的意义非常大。  
因为你终于可以对 AI 说出这种高质量指令：

> 本轮只改界面层样式，逻辑层和数据层不动。

这句话的价值，比很多长 prompt 都大。

### 3. 蓝图先行

写代码前先出蓝图，是 vibecoding 从“试玩”进入“工程化”的分水岭。

最简单的做法就是让 AI 先用 Mermaid 画出：

1. 用户路径
2. 模块依赖
3. 数据流转
4. 错误路径

你只要在开工前多花 10 分钟审蓝图，通常就能少掉后面几个小时的返工。

### 4. 规格驱动

模块拆了、分层清了、蓝图画了，还不够。

开工前还要把接口和字段写清楚，也就是所谓的规格驱动开发。

比如：

- 输入是什么
- 返回什么
- 字段类型是什么
- 什么情况报错
- 边界条件有哪些

只要规格清楚，后续无论换人、换模型、换实现方式，模块之间都不会轻易错位。

## 四、真正高效的 vibecoding，应该怎么做

我建议把流程压成 9 步：

1. 定北极星  
   一句话目标、目标用户、成功标准、第一版不做什么。

2. 立边界  
   数据红线、权限红线、敏感目录、哪些命令不能自动执行。

3. 拆 MVP  
   只保留第一次能交付价值的最小闭环。

4. 拆模块  
   让每个模块职责单一、可独立验证。

5. 做三层分层  
   确认界面层、逻辑层、数据层，明确本轮改动范围。

6. 先出蓝图  
   用 Mermaid 确认用户流、模块流、数据流、异常流。

7. 先写规格  
   把接口契约、字段约束、测试点写清楚。

8. 小步施工  
   单模块、单目标、单会话，配合 Git 小步提交。

9. 排错与复盘  
   先分层定位，再追踪数据流，最后只做最小修复。

这 9 步不是为了“显得专业”，而是为了让你在 AI 很强的时候，依然拥有工程上的稳定性。

## 五、很多准备工作，其实工具已经帮你做了

这也是很多个人开发者容易误判的地方。

不是所有项目都需要先写一大堆文档。  
因为很多 AI coding 工具，本来就已经内置了不少能力。

比如：

- `Cursor` 有项目规则、代码库索引、MCP、后台代理。
- `Claude Code` 有 `CLAUDE.md`、上下文压缩、subagents、hooks。
- `Aider` 有 repo map、模式切换、自动 lint/test、Git 回滚。
- `GitHub Copilot` 有 coding agent、code review、仓库级 instructions。
- `Continue` 有 rules、tool policies、MCP、多模型定制。

这意味着：

1. 既有项目通常不需要手写冗长的“代码库总览”。
2. 项目规范不必每轮都重复粘贴在 prompt 里。
3. 一部分任务可以异步后台执行，不必全塞进同一个对话。
4. 基础回滚和工具接入，也不一定都要你自己手工搭。

但与此同时，有些事情永远不能省：

- 产品目标
- 第一版边界
- 数据与权限红线
- 核心模块边界
- 验收标准
- 高风险接口约束

这些不是工具替你做不了，而是它不应该替你做。

## 六、个人开发者最实用的做法，不是全做，而是先做最小集

如果你现在就要开始一个项目，不需要先写一整本 PRD。

你只要先准备这 6 件事：

1. 一句话目标
2. 目标用户
3. 第一版只做什么
4. 第一版不做什么
5. 哪些区域是高风险禁区
6. 本轮只动哪些模块

然后就可以开始：

1. 让 AI 帮你拆 MVP
2. 让 AI 输出目录树和模块职责
3. 让 AI 做三层分层
4. 让 AI 先画蓝图
5. 让 AI 只实现当前子任务
6. 每做完一步就验证并提交

真正高效的 vibecoding，从来不是“让 AI 一次做完整个项目”，而是：

> 让 AI 在你的边界里持续稳定地产生正确的小结果。

## 七、最后一句话

这个时代最稀缺的能力，已经不是“会不会写代码”，而是：

你脑子里有没有一张清楚的系统蓝图，能把 AI 从一个会写代码的工具，变成一支真正受控、可靠、可扩展的产品施工队。

如果有，那么 AI 会让你快很多。  
如果没有，那么 AI 只会让你更快地混乱。

## 版本二：适合自己日常执行的超短 checklist 版

# Vibecoding 超短 Checklist

## A. 开工前 2 分钟

- [ ] 一句话目标写清楚了
- [ ] 目标用户写清楚了
- [ ] 第一版不做什么写清楚了
- [ ] 高风险区域列出来了
- [ ] 本轮只动哪些模块写清楚了

## B. 开工前 5 分钟

- [ ] 已经拆出 MVP，不是平台级需求
- [ ] 已经拆模块，不是一个大任务全塞给 AI
- [ ] 已经分清界面层 / 逻辑层 / 数据层
- [ ] 已经说明本轮允许改哪层，不允许改哪层
- [ ] 已经先让 AI 画蓝图，不是直接开写
- [ ] 已经写接口/字段/错误返回，不靠猜

## C. 施工中

- [ ] 一次只做一个子任务
- [ ] 一个会话只处理一个目标
- [ ] 只改允许改的文件
- [ ] AI 不允许顺手扩展
- [ ] 每做完一步立刻验证
- [ ] 每通过一步立刻提交 Git

## D. 报错时

- [ ] 先判断是哪一层出问题
- [ ] 再沿数据流追踪
- [ ] 不先让 AI 直接“修一下”
- [ ] 只修问题域，不全局乱改
- [ ] 修完后做最小回归验证

## E. 收尾前

- [ ] AI 已经做过一次自检
- [ ] 我自己复核过边界情况
- [ ] 旧功能影响点已经检查
- [ ] 最小验证集已经跑过
- [ ] 这次改动可以独立回滚
- [ ] 有必要的话把经验写进规则文件

## F. 绝不省略

- [ ] 产品目标
- [ ] 第一版边界
- [ ] 数据/权限/安全红线
- [ ] 核心模块边界
- [ ] 完成定义
- [ ] 高风险改动人工复核

## G. 每次都提醒自己

- [ ] AI 是施工队，不是总负责人
- [ ] 先目标，后编码
- [ ] 先边界，后实现
- [ ] 先蓝图，后施工
- [ ] 小步提交，可回滚
- [ ] 先定位，再修复
- [ ] 能做小闭环，就不要做大而全
